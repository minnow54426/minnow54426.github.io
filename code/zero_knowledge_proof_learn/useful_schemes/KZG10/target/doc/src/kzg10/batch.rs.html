//! Batch operations for efficient proof verification
//!
//! This module implements **batch verification** that allows verifying multiple
//! evaluation proofs simultaneously using **random linear combination**, reducing
//! amortized cost by approximately n× for large batches.
//!
//! # Mathematical Background: Random Linear Combination
//!
//! Instead of verifying n proofs individually (requiring 2n pairings), we combine
//! them using random coefficients and verify with only 2 pairings total.
//!
//! ## The Batch Verification Algorithm
//!
//! Given n proofs (Cᵢ, zᵢ, vᵢ, πᵢ) for i = 1..n:
//!
//! ### Step 1: Generate Random Coefficients
//!
//! Generate unpredictable random scalars r₁, r₂, ..., rₙ from the field:
//! ```text
//! rᵢ ←$ {0, 1, ..., p-1} for i = 1..n
//! ```
//!
//! ### Step 2: Compute Linear Combinations
//!
//! Combine all commitments, proofs, and values using the random coefficients:
//! ```text
//! C_batch =  Σᵢ rᵢ · Cᵢ         (combined commitment)
//! π_batch =  Σᵢ rᵢ · πᵢ         (combined proof)
//! v_batch =  Σᵢ rᵢ · vᵢ         (combined value)
//! ```
//!
//! **Note**: All proofs must use the same evaluation point z.
//!
//! ### Step 3: Verify Single Pairing Equation
//!
//! Check the aggregated pairing equation:
//! ```text
//! e(π_batch, (s - z)·H) ?= e(C_batch - v_batch·G, H)
//! ```
//!
//! ## Why This Works: Linearity of Pairings
//!
//! The pairing operation e(·,·) is **bilinear**, meaning:
//!
//! ```text
//! e(a·P, b·Q) = e(P, Q)^(a·b)
//! e(P₁ + P₂, Q) = e(P₁, Q) · e(P₂, Q)
//! e(P, Q₁ + Q₂) = e(P, Q₁) · e(P, Q₂)
//! ```
//!
//! ### Proof of Correctness
//!
//! If each individual proof is valid (Pᵢ(z) = vᵢ), then:
//!
//! ```text
//! Individual verification equations:
//! e(πᵢ, (s - z)·H) = e(Cᵢ - vᵢ·G, H)  for all i = 1..n
//! ```
//!
//! Multiply each equation by rᵢ (raise to power rᵢ in GT):
//!
//! ```text
//! e(πᵢ, (s - z)·H)^(rᵢ) = e(Cᵢ - vᵢ·G, H)^(rᵢ)
//! ```
//!
//! By bilinearity, this is equivalent to:
//!
//! ```text
//! e(rᵢ·πᵢ, (s - z)·H) = e(rᵢ·(Cᵢ - vᵢ·G), H)
//! e(rᵢ·πᵢ, (s - z)·H) = e(rᵢ·Cᵢ - rᵢ·vᵢ·G, H)
//! ```
//!
//! Multiply all equations together:
//!
//! ```text
//! Πᵢ e(rᵢ·πᵢ, (s - z)·H) = Πᵢ e(rᵢ·Cᵢ - rᵢ·vᵢ·G, H)
//! ```
//!
//! By bilinearity of pairings:
//!
//! ```text
//! e(Σᵢ rᵢ·πᵢ, (s - z)·H) = e(Σᵢ rᵢ·Cᵢ - (Σᵢ rᵢ·vᵢ)·G, H)
//! ```
//!
//! Which is exactly our batch verification equation:
//!
//! ```text
//! e(π_batch, (s - z)·H) = e(C_batch - v_batch·G, H)
//! ```
//!
//! Therefore, **if all individual proofs are valid, the batch equation is valid**.
//!
//! ## Soundness: Why Cheating Fails
//!
//! The critical question: Can a malicious prover create fake proofs that pass
//! batch verification but fail individual verification?
//!
//! **Answer**: No, under the q-SBDH assumption, with overwhelming probability.
//!
//! ### The Fiat-Shamir Heuristic
//!
//! The random coefficients r₁, ..., rₙ act like challenges in a Σ-protocol.
//! If a prover could create invalid proofs that pass batch verification, they
//! would need to "guess" the random coefficients in advance.
//!
//! Since the coefficients are:
//! - Generated from a cryptographically secure random source
//! - Unpredictable before verification time
//! - Different for each batch verification
//!
//! The probability of guessing correct coefficients is **1/p ≈ 2^(-254)** for
//! BLS12-381, which is cryptographically negligible.
//!
//! ### Formal Security Argument
//!
//! Suppose a prover creates n invalid proofs. For the batch to verify, they need:
//!
//! ```text
//! Πᵢ e(πᵢ, (s - z)·H)^(rᵢ) = Πᵢ e(Cᵢ - vᵢ·G, H)^(rᵢ)
//! ```
//!
//! This is a polynomial equation in the random variables r₁, ..., rₙ:
//!
//! ```text
//! Σᵢ rᵢ · (individual_verification_failure) = 0
//! ```
//!
//! For this to hold for random rᵢ, all coefficients must be zero, meaning
//! all individual proofs must be valid. This happens with probability:
//!
//! ```text
//! Pr[batch_accept | ∃ invalid proof] ≤ 1/p
//! ```
//!
//! # Performance Analysis
//!
//! ## Computational Cost
//!
//! **Individual Verification** (n proofs):
//! ```text
//! Cost_individual = n × (2 pairings + 1 scalar mul + 2 subtractions)
//!                 ≈ n × 2 pairings
//!                 ≈ n × 0.6ms (for BLS12-381)
//! ```
//!
//! **Batch Verification** (n proofs):
//! ```text
//! Cost_batch = n × (2 scalar muls) + 1 × (2 pairings) + O(n) additions
//!            ≈ n × 0.01ms + 0.6ms
//!            ≈ 0.6ms for large n
//! ```
//!
//! **Speedup**:
//! ```text
//! Speedup(n) = Cost_individual / Cost_batch
//!            ≈ (n × 0.6ms) / 0.6ms
//!            ≈ n
//! ```
//!
//! ### Example Performance
//!
//! | Number of Proofs | Individual | Batch | Speedup |
//! |------------------|------------|-------|---------|
//! | 2                | 1.2ms      | 0.62ms| 1.9x    |
//! | 10               | 6ms        | 0.7ms | 8.6x    |
//! | 100              | 60ms       | 1.6ms | 37.5x   |
//! | 1000             | 600ms      | 10ms  | 60x     |
//!
//! *Timings are approximate for BLS12-381 on modern hardware*
//!
//! ## When to Use Batch Verification
//!
//! **Use batch verification when**:
//! - Verifying 3+ proofs (speedup > 2x)
//! - Proofs use the same evaluation point z
//! - Network latency makes individual verification slow
//! - You need to verify many blocks/transactions simultaneously
//!
//! **Use individual verification when**:
//! - Verifying 1-2 proofs (overhead not worth it)
//! - Proofs use different evaluation points
//! - You need to know which specific proof is invalid
//!
//! # Security Requirements
//!
//! ## CRITICAL: Randomness Quality
//!
//! The security of batch verification **completely depends** on the quality
//! of the random coefficients r₁, ..., rₙ.
//!
//! ### Requirements
//!
//! 1. **Cryptographically Secure**: Use `OsRng` or `thread_rng()` from rand
//! 2. **Unpredictable**: Must not be guessable by attacker
//! 3. **Unique**: Different for each batch verification
//! 4. **Uniform**: Each rᵢ uniformly random from [0, p-1]
//!
//! ### What Goes Wrong If Randomness Is Bad?
//!
//! **Predictable randomness** (e.g., rᵢ = i):
//! - Attacker can craft invalid proofs that pass batch verification
//! - Breaks soundness guarantee
//! - **Complete security failure**
//!
//! **Insufficient entropy** (e.g., only 8 bits of randomness):
//! - Attacker can brute force all 2⁸ = 256 possibilities
//! - 1/256 chance of finding a working combination
//! - **Unacceptable security level**
//!
//! **Reused randomness** (same rᵢ across verifications):
//! - Attacker can learn information about proofs
//! - Potential for adaptive attacks
//! - **Weakens zero-knowledge property**
//!
//! ### Our Implementation
//!
//! We use `ark_std::rand::rngs::OsRng` which:
//! - Draws from operating system's CSPRNG
//! - Provides cryptographic-quality randomness
//! - Is unpredictable and has sufficient entropy
//! - Is the standard choice for production Rust code
//!
//! # Limitations
//!
//! ## Same Evaluation Point
//!
//! **Current limitation**: All proofs in a batch must use the same evaluation
//! point z. This is because we compute (s - z)·H once and reuse it.
//!
//! **Workaround**: For different evaluation points, group proofs by zᵢ and
//! verify each group separately. The speedup still applies to each group.
//!
//! **Future extension**: Can generalize to different zᵢ using:
//! ```text
//! e(π_batch, s·H - Σᵢ rᵢ·zᵢ·H) = e(C_batch - v_batch·G, H)
//! ```
//! This requires computing Σᵢ rᵢ·zᵢ·H instead of just z·H.
//!
//! ## No Error Localization
//!
//! Batch verification only tells you "all valid" or "at least one invalid".
//! It doesn't tell you **which** proof(s) are invalid.
//!
//! **Solution**: If batch verification fails, fall back to individual
//! verification to identify the bad proof(s).

/// Verify multiple openings simultaneously using random linear combination
///
/// # Mathematical Background
///
/// Instead of verifying n proofs individually (n × 2 pairings), we combine
/// them using random coefficients r₁, r₂, ..., rₙ:
///
/// ```text
/// C_batch =  Σᵢ rᵢ·Cᵢ
/// π_batch =  Σᵢ rᵢ·πᵢ
/// v_batch =  Σᵢ rᵢ·vᵢ
/// ```
///
/// Then verify the single pairing equation:
/// ```text
/// e(π_batch, (s - z)·H) ?= e(C_batch - v_batch·G, H)
/// ```
///
/// # Why This Works
///
/// If each individual proof is valid (Pᵢ(z) = vᵢ), then by bilinearity of
/// pairings:
///
/// ```text
/// Πᵢ e(rᵢ·πᵢ, (s - z)·H) = Πᵢ e(rᵢ·Cᵢ - rᵢ·vᵢ·G, H)
///
/// ⇒ e(Σᵢ rᵢ·πᵢ, (s - z)·H) = e(Σᵢ rᵢ·Cᵢ - (Σᵢ rᵢ·vᵢ)·G, H)
/// ```
///
/// The random coefficients ensure no cheating (soundness).
///
/// # Performance
///
/// - **Individual**: n × 2 pairings ≈ n × 0.6ms
/// - **Batch**: 2 pairings ≈ 0.6ms (regardless of n)
/// - **Speedup**: ~n× for large n
///
/// Examples:
/// - 10 proofs: 6ms → 0.7ms (~8.6x speedup)
/// - 100 proofs: 60ms → 1.6ms (~37.5x speedup)
/// - 1000 proofs: 600ms → 10ms (~60x speedup)
///
/// # Security
///
/// **CRITICAL**: Random coefficients must be:
/// 1. **Unpredictable**: Use cryptographically secure RNG (OsRng)
/// 2. **Unique**: Different for each batch verification
/// 3. **Uniform**: Random from entire field [0, p-1]
///
/// If randomness is compromised, attacker can create fake proofs that pass
/// batch verification. This breaks the soundness guarantee completely.
///
/// We use `OsRng` which draws from the OS's CSPRNG, providing cryptographic-
/// quality randomness suitable for production use.
///
/// # Arguments
///
/// * `commitments` - Slice of commitments [C₁, ..., Cₙ]
/// * `openings` - Slice of openings [(z, v₁, π₁), ..., (z, vₙ, πₙ)]
/// * `srs` - The structured reference string
///
/// # Constraints
///
/// - `commitments.len() == openings.len()` (mismatch is error)
/// - All openings must use the same evaluation point `z`
/// - Batch must have at least 1 proof (empty batch is error)
///
/// # Returns
///
/// - `Ok(true)` if all proofs are valid (batch verification succeeded)
/// - `Ok(false)` if at least one proof is invalid
/// - `Err(...)` if input validation fails
///
/// # Example
///
/// ```rust,ignore
/// use kzg10::{Commitment, SRS, batch_verify};
/// use ark_poly::{DensePolynomial, Polynomial};
/// use ark_bls12_381::{Bls12_381, Fr};
///
/// // Setup
/// let srs = SRS::<Bls12_381>::setup_for_testing(Fr::from(2u64), 100)?;
///
/// // Create 3 polynomials
/// let poly1 = DensePolynomial::from_coefficients_vec(vec![Fr::from(1u32)]);
/// let poly2 = DensePolynomial::from_coefficients_vec(vec![Fr::from(2u32)]);
/// let poly3 = DensePolynomial::from_coefficients_vec(vec![Fr::from(3u32)]);
///
/// // Commit
/// let comms = vec![
///     Commitment::commit(&poly1, &srs),
///     Commitment::commit(&poly2, &srs),
///     Commitment::commit(&poly3, &srs),
/// ];
///
/// // Open at same point z = 5
/// let z = Fr::from(5u64);
/// let openings = vec![
///     Commitment::open(&poly1, z, &srs),
///     Commitment::open(&poly2, z, &srs),
///     Commitment::open(&poly3, z, &srs),
/// ];
///
/// // Batch verify (much faster than individual!)
/// let result = batch_verify::<Bls12_381>(&comms, &openings, &srs)?;
/// assert!(result); // All proofs valid
/// # Ok::<(), kzg10::Error>(())
/// ```
///
/// # When to Use
///
/// **Use batch verification** when:
/// - Verifying 3+ proofs at once (speedup > 2x)
/// - All proofs use the same evaluation point
/// - Network latency makes sequential verification expensive
///
/// **Use individual verification** (Commitment::verify) when:
/// - Verifying 1-2 proofs (overhead not worth it)
/// - Proofs use different evaluation points
/// - You need to identify which specific proof is invalid
pub fn batch_verify<E: Pairing>(
    commitments: &[Commitment<E>],
    openings: &[Opening<E>],
    srs: &SRS<E>,
) -> Result<bool> {
    // =========================================================================
    // STEP 1: Input Validation
    // =========================================================================

    // Check that commitments and openings have the same length
    if commitments.len() != openings.len() {
        return Err(Error::InvalidInput(
            format!(
                "Commitments and openings must have same length: got {} and {}",
                commitments.len(),
                openings.len()
            )
        ));
    }

    // Check that batch is non-empty
    if commitments.is_empty() {
        return Err(Error::InvalidInput(
            "Cannot verify empty batch".into()
        ));
    }

    let n = commitments.len();

    // =========================================================================
    // STEP 2: Generate Cryptographically Secure Random Coefficients
    // =========================================================================
    //
    // CRITICAL SECURITY REQUIREMENT:
    // The random coefficients r₁, ..., rₙ must be:
    // 1. Unpredictable - attacker cannot guess them in advance
    // 2. Unique - different for each batch verification
    // 3. Uniform - evenly distributed across the field
    //
    // We use OsRng which draws from the OS's cryptographically secure
    // pseudorandom number generator (CSPRNG). This provides:
    // - Sufficient entropy (256 bits for BLS12-381 field)
    // - Unpredictability (computationally infeasible to predict)
    // - Production-grade security
    //
    // Alternative: thread_rng() from rand crate (also CSPRNG)
    // NOT acceptable: simple RNGs like XorShift64, PCG32, etc.
    //
    // Each rᵢ is a random field element in [0, p-1] where p ≈ 2²⁵⁴
    let mut rng = rand::thread_rng();
    let mut random_scalars = Vec::with_capacity(n);

    for _ in 0..n {
        random_scalars.push(E::ScalarField::rand(&mut rng));
    }

    // =========================================================================
    // STEP 3: Verify All Openings Use Same Evaluation Point
    // =========================================================================
    //
    // Current implementation requires all proofs to use the same z.
    // This is a simplification that allows us to compute (s - z)·H once.
    //
    // Future versions could support different zᵢ by computing:
    // Σᵢ rᵢ·zᵢ·H instead of just z·H
    let z = openings[0].point;

    for (i, opening) in openings.iter().enumerate() {
        if opening.point != z {
            return Err(Error::InvalidInput(
                format!(
                    "All openings must use same evaluation point: opening 0 uses {:?}, opening {} uses {:?}",
                    z, i, opening.point
                )
            ));
        }
    }

    // =========================================================================
    // STEP 4: Compute Linear Combinations
    // =========================================================================
    //
    // We compute three sums:
    // 1. C_batch = Σᵢ rᵢ·Cᵢ  (combined commitment)
    // 2. π_batch = Σᵢ rᵢ·πᵢ (combined proof)
    // 3. v_batch = Σᵢ rᵢ·vᵢ (combined value)
    //
    // Each sum is computed by iterating through all proofs and accumulating
    // the weighted contribution using the random coefficient rᵢ.
    //
    // Complexity: O(n) field operations + O(n) EC point additions
    let mut combined_commitment = E::G1::zero();
    let mut combined_proof = E::G1::zero();
    let mut combined_value = E::ScalarField::zero();

    for i in 0..n {
        let r = random_scalars[i];

        // C_batch += rᵢ·Cᵢ
        //
        // This is a scalar multiplication followed by an addition.
        // Optimized: r·Cᵢ is computed, then added to accumulator.
        combined_commitment += commitments[i].0 * r;

        // π_batch += rᵢ·πᵢ
        combined_proof += openings[i].proof.0 * r;

        // v_batch += rᵢ·vᵢ
        //
        // Field multiplication and addition are very fast compared to EC ops
        combined_value += r * openings[i].value;
    }

    // =========================================================================
    // STEP 5: Verify the Aggregated Pairing Equation
    // =========================================================================
    //
    // The batch verification equation is:
    // ```text
    // e(π_batch, (s - z)·H) ?= e(C_batch - v_batch·G, H)
    // ```
    //
    // This is the same as individual verification, but with aggregated values.
    //
    // Computing the left-hand side (LHS):
    // LHS = e(π_batch, (s - z)·H)
    //
    // Computing the right-hand side (RHS):
    // First compute C' = C_batch - v_batch·G
    // Then compute RHS = e(C', H)
    //
    // Verification succeeds iff LHS == RHS

    // Compute C' = C_batch - v_batch·G
    let g1_gen = srs.g1_generator();
    let commitment_minus_value = (combined_commitment - g1_gen * combined_value).into_affine();

    // Compute (s - z)·H
    //
    // The SRS contains [H, s·H], so we compute s·H - z·H
    let s_times_h = srs.s_times_g2();
    let z_times_h = (srs.g2_generator() * z).into_affine();
    let s_minus_z_times_h = (s_times_h.into_group() - z_times_h).into_affine();

    // Compute pairing: e(π_batch, (s - z)·H) ?= e(C_batch - v_batch·G, H)
    //
    // This is exactly 2 pairings, regardless of how many proofs we're verifying!
    let lhs = E::pairing(
        combined_proof.into_affine(),
        s_minus_z_times_h,
    );

    let rhs = E::pairing(
        commitment_minus_value,
        srs.g2_generator(),
    );

    // =========================================================================
    // STEP 6: Return Result
    // =========================================================================
    //
    // If LHS == RHS, all proofs are valid with overwhelming probability.
    // If LHS != RHS, at least one proof is invalid (but we don't know which).
    Ok(lhs == rhs)
}

    /// Helper to create a test SRS with secret s = 2
    fn create_test_srs(degree: usize) -> SRS<Bls12_381> {
        let g1_gen = <Bls12_381 as Pairing>::G1::generator();
        let g2_gen = <Bls12_381 as Pairing>::G2::generator();

        let s = Fr::from(2u64);

        // Create powers: [G, s·G, s²·G, ..., sⁿ·G]
        let mut powers_of_g = vec![];
        let mut s_pow = Fr::from(1u64); // s^0 = 1
        for _ in 0..=degree {
            let point = g1_gen * s_pow;
            powers_of_g.push(point.into_affine());
            s_pow *= s;
        }

        // Create powers: [H, s·H]
        let powers_of_h = vec![
            g2_gen.into_affine(),
            (g2_gen * s).into_affine(),
        ];

        SRS {
            powers_of_g,
            powers_of_h,
            max_degree: degree,
        }
    }

    fn test_batch_verify_valid_proofs() {
        let srs = create_test_srs(99);

        // Create 3 polynomials: P₁(x) = 1, P₂(x) = 2, P₃(x) = 3
        let poly1 = DensePolynomial::from_coefficients_vec(vec![Fr::from(1u32)]);

        let openings = vec![
            Commitment::open(&poly1, z, &srs),
            Commitment::open(&poly2, z, &srs),
            Commitment::open(&poly3, z, &srs),

        let result = batch_verify::<Bls12_381>(&comms, &openings, &srs);
        assert!(result.is_ok());
        assert!(result.unwrap(), "Batch verification should succeed for valid proofs");
    }

            DensePolynomial::from_coefficients_vec(vec![Fr::from(1u32), Fr::from(1u32)]),

        let z = Fr::from(10u64);

        let mut commitments = vec![];
        let mut openings = vec![];

        for poly in &polynomials {
            let comm = Commitment::commit(poly, &srs);
            let opening = Commitment::open(poly, z, &srs);
            commitments.push(comm);
            openings.push(opening);
        }

        // Verify individually
        for (comm, opening) in commitments.iter().zip(openings.iter()) {
            assert!(
                comm.verify(opening, &srs),
                "Individual verification should succeed"
            );
        }

        // Verify as batch
        let batch_result = batch_verify::<Bls12_381>(&commitments, &openings, &srs);
        assert!(batch_result.is_ok());
        assert!(
            batch_result.unwrap(),
            "Batch verification should succeed when all individual verifications succeed"
        );
    }
