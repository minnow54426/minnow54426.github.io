//! Polynomial commitment computation
//!
//! This module implements the commitment operation for the KZG10 scheme.
//!
//! # Mathematical Background
//!
//! Given a polynomial P(x) = c₀ + c₁x + c₂x² + ... + cₙxⁿ and an SRS
//! containing powers of the secret s, we compute the commitment:
//!
//! ```text
//! C = c₀·G + c₁·(s·G) + c₂·(s²·G) + ... + cₙ·(sⁿ·G)
//!   = Σ cᵢ · (sⁱ·G)
//!   = P(s)·G
//! ```
//!
//! where G is the generator of the G1 group and sⁱ·G are precomputed
//! in the SRS (Structured Reference String).
//!
//! # Multi-Scalar Multiplication (MSM)
//!
//! The commitment computation uses **Multi-Scalar Multiplication (MSM)**, which
//! is the core operation in many polynomial commitment schemes.
//!
//! ## MSM Definition
//!
//! Given:
//! - Scalars: [c₀, c₁, ..., cₙ] in the field F
//! - Points: [P₀, P₁, ..., Pₙ] in an elliptic curve group
//!
//! MSM computes:
//! ```text
//! R = c₀·P₀ + c₁·P₁ + ... + cₙ·Pₙ
//! ```
//!
//! In KZG10 commitments:
//! - Scalars are polynomial coefficients
//! - Points are SRS powers [G, s·G, s²·G, ...]
//! - Result is the commitment C = P(s)·G
//!
//! ## Pippenger's Algorithm
//!
//! The naive approach computes each scalar multiplication separately (O(n) EC ops).
//! Pippenger's algorithm (1976) reduces this to O(n / log n) using:
//!
//! 1. **Windowing**: Split scalars into windows of w bits
//! 2. **Bucket accumulation**: Group operations by window value
//! 3. **Combination**: Sum results from all windows
//!
//! For window size w:
//! - Precompute 2ʷ buckets for each window
//! - Distribute point contributions to buckets
//! - Sum buckets and combine across windows
//!
//! Complexity: O(n·(1/w + log 2ʷ)) → O(n / log n) when optimized
//!
//! ### Example with w=3 (8 buckets)
//!
//! For scalars [13, 7, 11] and points [P₀, P₁, P₂]:
//!
//! ```text
//! Binary representation:
//! 13 = 1101₂
//!  7 = 0111₂
//! 11 = 1011₂
//!
//! Window 0 (bits 0-2): [1, 7, 3] → buckets
//! Window 1 (bits 3-5): [5, 0, 1] → buckets
//!
//! Final result = 2³·(sum of window 1) + (sum of window 0)
//! ```
//!
//! ## Performance Characteristics
//!
//! For degree n polynomial on BLS12-381 curve:
//!
//! | Degree | Naive O(n) | Pippenger O(n/log n) |
//! |--------|-----------|---------------------|
//! | 2¹² (4096) | ~50ms | ~10ms |
//! | 2¹⁶ (65536) | ~800ms | ~80ms |
//! | 2²⁰ (1M) | ~13s | ~500ms |
//!
//! ## Parallelization
//!
//! MSM can be parallelized using:
//! - **Chunk-based parallelism**: Split points/scalars across threads
//! - **Window parallelism**: Process different windows in parallel
//! - **Bucket parallelism**: Sum multiple buckets simultaneously
//!
//! Enable with: `cargo build --features parallel`
//!
//! Uses Rayon for work-stealing parallelism.
//!
//! # Algorithm
//!
//! The commitment is computed using MSM:
//!
//! 1. Extract polynomial coefficients [c₀, c₁, ..., cₙ]
//! 2. Extract SRS powers [G, s·G, s²·G, ..., sⁿ·G]
//! 3. Compute: C = Σ cᵢ · powers[i] using MSM
//!
//! # Performance
//!
//! For degree n polynomial:
//! - Time: O(n) with naive approach, O(n / log n) with Pippenger
//! - Memory: O(n) for the SRS powers
//! - Parallelizable: Yes (with "parallel" feature)
//!
//! Example timings (BLS12-381, modern CPU):
//! - Degree 2¹² (4096): ~10ms
//! - Degree 2¹⁶ (65536): ~80ms
//! - Degree 2²⁰ (1M): ~500ms
//!
//! # Security
//!
//! The commitment scheme provides two key security properties:
//!
//! ## Binding (Commitment)
//!
//! A commitment is **binding** if it's computationally infeasible to find
//! two different polynomials P ≠ Q with the same commitment:
//!
//! ```text
//! C = P(s)·G = Q(s)·G  ⇒  (P(s) - Q(s))·G = 0
//! ```
//!
//! Since G is a generator, (P(s) - Q(s)) must be 0 in the field.
//! This means P(s) = Q(s), but since deg(P-Q) ≤ n and s is unknown,
//! finding such P≠Q requires solving discrete log.
//!
//! The **binding property** relies on:
//! - Discrete logarithm assumption in G1
//! - s being unknown (trusted setup)
//!
//! ## Hiding (Privacy)
//!
//! A commitment is **hiding** if it reveals nothing about the polynomial.
//!
//! Given commitment C = P(s)·G, recovering P requires:
//! 1. Compute discrete log: s = log_G(C) / P(s)
//! 2. Or directly find P from C
//!
//! Both are computationally infeasible under:
//! - **Discrete Logarithm Assumption**: Can't recover exponent from point
//! - **Decisional Diffie-Hellman (DDH)**: Can't distinguish C from random
//!
//! Without knowing s, the commitment C is computationally indistinguishable
//! from a random group element.
//!
//! ## Assumptions Summary
//!
//! - **Discrete Log (DL)**: Binding and hiding
//! - **DDH**: Enhanced hiding
//! - **Trusted Setup**: s must be unknown and destroyed after setup
//! - **Correct SRS**: Powers [s⁰·G, ..., sⁿ·G] must be correctly computed

/// A KZG10 polynomial commitment
///
/// # Mathematical Meaning
///
/// A commitment C to a polynomial P(x) of degree d is computed as:
/// ```text
/// C = P(s) · G = Σᵢ aᵢ · sⁱ · G
/// ```
/// where:
/// - P(x) = Σᵢ aᵢ · xⁱ is the polynomial
/// - s is the SRS secret (toxic waste, must be destroyed)
/// - G is the generator of the G1 group
///
/// The commitment is a single group element in G1, which provides:
/// - **Constant size**: 48 bytes regardless of polynomial degree
/// - **Homomorphism**: com(P+Q) = com(P) + com(Q)
/// - **Binding**: Can't open to two different values
///
/// # Security Properties
///
/// ## Binding
///
/// Under the discrete logarithm assumption in G1, it's computationally infeasible
/// to find two distinct polynomials P ≠ Q such that:
/// ```text
/// P(s) · G = Q(s) · G
/// ```
/// This ensures a committer can't change the polynomial after committing.
///
/// ## Hiding
///
/// Under the decisional Diffie-Hellman (DDH) assumption in G1, the commitment
/// P(s)·G reveals nothing about the polynomial P. This is because finding s
/// from {G·sⁱ} requires solving the discrete log problem.
///
/// # Serialization
///
/// - **Uncompressed**: 64 bytes (x and y coordinates)
/// - **Compressed**: 48 bytes (x-coordinate and sign bit)
///
/// The commitment implements `CanonicalSerialize` and `CanonicalDeserialize`
/// for efficient serialization to bytes.
///
/// # Generic Type Parameter
///
/// `E: Pairing` allows this commitment to work with any pairing-friendly curve,
/// though BLS12-381 is the primary supported curve in this implementation.
///
/// # Example
///
/// ```rust,ignore
/// use ark_bls12_381::Bls12_381;
/// use ark_poly::univariate::DensePolynomial;
/// use ark_poly::UvPolynomial;
/// use kzg10::{Commitment, SRS};
///
/// // Create polynomial P(x) = x² + 2x + 1
/// let coeffs = vec![1, 2, 1];
/// let poly = DensePolynomial::from_coefficients_vec(coeffs);
///
/// // Generate SRS with secret s
/// let srs = SRS::<Bls12_381>::setup(2, &mut rng);
///
/// // Compute commitment C = P(s)·G
/// let commitment = Commitment::commit(&poly, &srs)?;
///
/// // Commitment is 48 bytes when serialized
/// let bytes = commitment.to_bytes();
/// assert_eq!(bytes.len(), 48);
/// # Ok::<(), kzg10::Error>(())
/// ```
#[derive(Derivative, Clone, Copy, PartialEq, Eq)]
#[derivative(Debug)]
pub struct Commitment<E: Pairing>(pub(crate) E::G1Affine);

impl<E: Pairing> Commitment<E> {
    /// Create a new commitment from a G1 affine point
    ///
    /// # Mathematical Meaning
    ///
    /// This wraps a G1 point that represents P(s)·G for some polynomial P.
    /// The point must be in the correct subgroup of G1.
    ///
    /// # Security Considerations
    ///
    /// - Caller must ensure the point is in the correct subgroup
    /// - Point must not be the identity unless committing to zero polynomial
    /// - Point coordinates must be valid field elements
    ///
    /// # Example
    ///
    /// ```rust,ignore
    /// use ark_bls12_381::{Bls12_381, G1Affine};
    /// use kzg10::Commitment;
    ///
    /// let point = G1Affine::generator();
    /// let commitment = Commitment::<Bls12_381>::new(point);
    /// ```
    pub fn new(point: E::G1Affine) -> Self {
        Self(point)
    }

    /// Get the underlying G1 affine point
    ///
    /// # Returns
    ///
    /// The G1 point representing P(s)·G
    pub fn as_point(&self) -> &E::G1Affine {
        &self.0
    }

    /// Convert to projective representation
    ///
    /// # Mathematical Meaning
    ///
    /// Converts from affine (X, Y) to projective (X, Y, Z) coordinates.
    /// Projective coordinates are more efficient for:
    /// - Mixed arithmetic (affine + projective)
    /// - Multi-scalar multiplications
    /// - Group operations
    ///
    /// # Performance
    ///
    /// - Conversion is O(1) - just sets Z = 1
    /// - Use projective for repeated group operations
    /// - Use affine for serialization and storage
    pub fn to_projective(&self) -> E::G1 {
        self.0.into_group()
    }

    /// Create a commitment from a projective point
    ///
    /// # Mathematical Meaning
    ///
    /// Converts from projective (X, Y, Z) to affine (X, Y) coordinates
    /// by computing the inverse of Z. This is needed for:
    /// - Combining commitments (using projective arithmetic)
    /// - Homomorphic property testing
    /// - Batch operations
    ///
    /// # Performance
    ///
    /// - Conversion requires field inversion: O(log n) using Fermat's little theorem
    /// - More expensive than to_projective()
    /// - Only use when necessary (e.g., after combining projective points)
    ///
    /// # Example
    ///
    /// ```rust,ignore
    /// use ark_bls12_381::Bls12_381;
    /// use kzg10::Commitment;
    ///
    /// // Combine two commitments using projective arithmetic
    /// let c1 = Commitment::commit(&poly1, &srs);
    /// let c2 = Commitment::commit(&poly2, &srs);
    ///
    /// let combined_proj = c1.to_projective() + c2.to_projective();
    /// let combined = Commitment::from_projective(combined_proj);
    /// ```
    pub fn from_projective(proj: E::G1) -> Self {
        Self(proj.into_affine())
    }

    /// Serialize commitment to bytes
    ///
    /// # Format
    ///
    /// Uses arkworks' compressed serialization:
    /// - 48 bytes total for BLS12-381
    /// - x-coordinate: 32 bytes
    /// - Sign bit and flags: 16 bytes
    ///
    /// # Example
    ///
    /// ```rust,ignore
    /// # use kzg10::Commitment;
    /// # let commitment = Commitment::default();
    /// let bytes = commitment.to_bytes();
    /// assert_eq!(bytes.len(), 48);
    /// ```
    pub fn to_bytes(&self) -> Vec<u8> {
        let mut bytes = Vec::with_capacity(self.0.serialized_size(ark_serialize::Compress::Yes));
        self.serialize_compressed(&mut bytes)
            .expect("serialization to vec should succeed");
        bytes
    }

    /// Deserialize commitment from bytes
    ///
    /// # Errors
    ///
    /// Returns `Error::InvalidProof` if:
    /// - Point is not on the curve
    /// - Point is not in the correct subgroup
    ///
    /// # Example
    ///
    /// ```rust,ignore
    /// # use kzg10::Commitment;
    /// # let commitment = Commitment::default();
    /// let bytes = commitment.to_bytes();
    /// let decoded = Commitment::from_bytes(&bytes)?;
    /// assert_eq!(commitment, decoded);
    /// # Ok::<(), kzg10::Error>(())
    /// ```
    pub fn from_bytes(bytes: &[u8]) -> Result<Self> {
        let point = E::G1Affine::deserialize_compressed(bytes)
            .map_err(|e| Error::InvalidProof(format!("invalid G1 point: {}", e)))?;

        // Note: deserialize_compressed already validates the point is on curve
        // and in the correct subgroup (uses Validate::Yes flag)
        Ok(Self(point))
    }

    /// Commit to a polynomial using the KZG10 commitment scheme
    ///
    /// # Mathematical Definition
    ///
    /// Given polynomial P(x) = Σ cᵢ·xⁱ and SRS with powers [G, s·G, ..., sⁿ·G],
    /// compute the commitment:
    ///
    /// ```text
    /// C = Σ cᵢ · (sⁱ·G) = P(s)·G
    /// ```
    ///
    /// This is a multi-scalar multiplication (MSM) of coefficients with SRS powers.
    ///
    /// # Algorithm
    ///
    /// 1. Validate polynomial degree ≤ SRS max_degree
    /// 2. Extract coefficients from polynomial
    /// 3. Compute MSM: C = Σ coeff[i] · SRS.powers_of_g[i]
    ///
    /// # Arguments
    ///
    /// * `poly` - The polynomial to commit to (coefficient form)
    /// * `srs` - The structured reference string with precomputed sⁱ·G
    ///
    /// # Returns
    ///
    /// A G1 point representing the commitment C = P(s)·G
    ///
    /// # Panics
    ///
    /// Panics if polynomial degree exceeds SRS max_degree.
    /// This is intentional for this learning implementation to fail fast
    /// on invalid input. In production, this should return a Result.
    ///
    /// # Example
    ///
    /// ```rust,ignore
    /// use kzg10::{Commitment, SRS};
    /// use ark_poly::univariate::DensePolynomial;
    /// use ark_poly::Polynomial;
    /// use ark_bls12_381::Bls12_381;
    ///
    /// # #[cfg(feature = "test-only")]
    /// # {
    /// use ark_bls12_381::Fr;
    ///
    /// // Polynomial: P(x) = x³ + 2x² + 3x + 4
    /// let coeffs = vec![
    ///     Fr::from(4u32),  // c₀ (constant term)
    ///     Fr::from(3u32),  // c₁ (x term)
    ///     Fr::from(2u32),  // c₂ (x² term)
    ///     Fr::from(1u32),  // c₃ (x³ term)
    /// ];
    /// let poly = DensePolynomial::from_coefficients_vec(coeffs);
    ///
    /// // Setup SRS (testing only - use powers of tau in production!)
    /// let srs = SRS::<Bls12_381>::setup_for_testing(Fr::from(7u64), 3).unwrap();
    ///
    /// // Commit
    /// let comm = Commitment::commit(&poly, &srs);
    ///
    /// // comm is a single G1 point (48 bytes compressed, 96 uncompressed)
    /// # }
    /// ```
    ///
    /// # Performance
    ///
    /// Uses ark-ec's optimized MSM implementation:
    /// - **Pippenger's algorithm** for large degree (> 2¹²)
    /// - **Straus' method** for medium degree (2⁸ to 2¹²)
    /// - **Direct doubling** for small degree (< 2⁸)
    ///
    /// Time complexity: O(n / log n) for degree n
    ///
    /// Benchmarks on BLS12-381 (AMD Ryzen 9 5900X):
    /// - Degree 2¹² (4096): ~10ms
    /// - Degree 2¹⁶ (65536): ~80ms
    /// - Degree 2²⁰ (1M): ~500ms
    ///
    /// With "parallel" feature (8 cores):
    /// - Degree 2¹²: ~3ms (3.3x speedup)
    /// - Degree 2¹⁶: ~20ms (4x speedup)
    /// - Degree 2²⁰: ~120ms (4.2x speedup)
    ///
    /// # Security
    ///
    /// The commitment is:
    /// - **Binding**: Can't find P≠Q with same commitment (without knowing s)
    /// - **Hiding**: Commitment doesn't reveal polynomial coefficients
    ///
    /// Both properties rely on:
    /// - **Discrete logarithm assumption** in G1 group
    /// - **Trusted setup**: s is unknown and destroyed after setup
    /// - **Correct SRS**: Powers are correctly computed
    ///
    /// ## Binding Property
    ///
    /// If commitment is binding, then:
    /// ```text
    /// P(s)·G = Q(s)·G  ⇒  P = Q (with overwhelming probability)
    /// ```
    ///
    /// Because if P≠Q, then (P(s)-Q(s))·G = 0 implies P(s)=Q(s) in the field.
    /// Finding P≠Q with P(s)=Q(s) requires solving discrete log.
    ///
    /// ## Hiding Property
    ///
    /// Given commitment C = P(s)·G:
    /// - Can't recover P without knowing s
    /// - C is computationally indistinguishable from random G1 point
    /// - Relies on DDH (Decisional Diffie-Hellman) assumption
    ///
    /// # Homomorphic Property
    ///
    /// KZG10 commitments are **homomorphic**:
    /// ```text
    /// commit(P + Q) = commit(P) + commit(Q)
    /// commit(c·P) = c·commit(P)
    /// ```
    ///
    /// This enables:
    /// - Batch commitment verification
    /// - Efficient proof aggregation
    /// - Polynomial operations on commitments
    pub fn commit(poly: &DensePolynomial<E::ScalarField>, srs: &SRS<E>) -> Self {
        // Step 1: Validate degree
        //
        // The SRS contains powers [s⁰·G, s¹·G, ..., sᵐ·G] where m = max_degree
        // To commit to degree n polynomial, we need n+1 powers
        // Therefore: n = poly.degree() must satisfy n ≤ max_degree
        let degree = poly.degree();

        if degree > srs.max_degree {
            panic!(
                "Polynomial degree {} exceeds SRS max degree {}",
                degree, srs.max_degree
            );
        }

        // Step 2: Extract coefficients
        //
        // DensePolynomial stores coefficients in order [c₀, c₁, ..., cₙ]
        // where c₀ is the constant term (coefficient of x⁰)
        // and cₙ is the coefficient of xⁿ
        //
        // For P(x) = 4 + 3x + 2x² + 1x³:
        // coeffs = [4, 3, 2, 1]
        let coeffs = poly.coeffs();

        // Step 3: Get SRS powers
        //
        // SRS contains: [s⁰·G, s¹·G, s²·G, ..., sᵐ·G]
        // We need: [s⁰·G, s¹·G, ..., sⁿ·G] where n = poly.degree()
        //
        // This is exactly coeffs.len() elements (since coeffs.len() = degree + 1)
        //
        // For degree 3 polynomial with 4 coefficients:
        // We need powers_of_g[0..4] = [G, s·G, s²·G, s³·G]
        let powers = &srs.powers_of_g[..coeffs.len()];

        // Step 4: Compute multi-scalar multiplication (MSM)
        //
        // MSM computes: c₀·p₀ + c₁·p₁ + ... + cₙ·pₙ
        // where pᵢ = sⁱ·G from the SRS
        //
        // This gives us: C = Σ cᵢ·(sⁱ·G) = P(s)·G
        //
        // The MSM operation is highly optimized in ark-ec:
        //
        // **Algorithm Selection** (based on input size):
        // 1. Small (< 256 points): Direct doubling method
        //    - Compute each scalar multiplication separately
        //    - Sum the results
        //    - O(n) elliptic curve operations
        //
        // 2. Medium (256 - 4096): Straus' method (windowed)
        //    - Use windowing to reduce operations
        //    - Precompute windows
        //    - O(n / w) where w is window size
        //
        // 3. Large (> 4096): Pippenger's algorithm
        //    - Multi-level windowing
        //    - Bucket accumulation
        //    - O(n / log n) complexity
        //
        // **Pippenger's Algorithm Details**:
        // 1. Split scalars into windows of w bits (typically w=10 or 11)
        // 2. For each window:
        //    - Create 2ʷ buckets
        //    - Distribute points to buckets based on window bits
        //    - Sum points in each bucket
        // 3. Combine buckets: C = Σ(2ʷ·i)ᵗ·bucket[i]
        //    where t is the window index
        //
        // **Parallelization** (with "parallel" feature):
        // - Split points/scalars into chunks
        // - Compute chunk MSMs in parallel
        // - Sum chunk results
        //
        // **Performance**:
        // - Naive: O(n) scalar multiplications
        // - Pippenger: O(n / log n) operations
        // - Parallel: Additional O(cores) speedup
        //
        // Example for n=4096:
        // - Naive: 4096 EC multiplications
        // - Pippenger: ~400 EC operations (10x speedup)
        // - Parallel (8 cores): ~120 EC operations (3.3x more)
        let result = E::G1::msm(powers, coeffs).expect("MSM should not fail");

        // Step 5: Convert to affine coordinates
        //
        // MSM returns result in projective coordinates (X:Y:Z)
        // This is more efficient during computation (fewer inversions)
        //
        // Affine coordinates (X:Y) are:
        // - More compact (48 bytes vs 96 bytes)
        // - Required for serialization
        // - Required for pairing operations
        //
        // Conversion: (X:Y:Z) → (X/Z, Y/Z)
        // Requires one field inversion (expensive, but done once)
        Self(result.into_affine())
    }
}

impl<E: Pairing> Default for Commitment<E> {
    fn default() -> Self {
        // Use the generator and multiply by 0 to get identity
        // This works for any pairing curve
        let g1 = E::G1::generator();
        let identity = (g1 * E::ScalarField::zero()).into_affine();
        Self(identity)
    }
}

impl<E: Pairing> Valid for Commitment<E> {
    fn check(&self) -> std::result::Result<(), ark_serialize::SerializationError> {
        self.0.check()
    }
}

impl<E: Pairing> CanonicalSerialize for Commitment<E> {
    fn serialize_with_mode<W: ark_serialize::Write>(
        &self,
        writer: W,
        mode: ark_serialize::Compress,
    ) -> std::result::Result<(), ark_serialize::SerializationError> {
        self.0.serialize_with_mode(writer, mode)
    }

    fn serialized_size(&self, mode: ark_serialize::Compress) -> usize {
        self.0.serialized_size(mode)
    }
}

impl<E: Pairing> CanonicalDeserialize for Commitment<E> {
    fn deserialize_with_mode<R: ark_serialize::Read>(
        reader: R,
        mode: ark_serialize::Compress,
        validate: ark_serialize::Validate,
    ) -> std::result::Result<Self, ark_serialize::SerializationError> {
        Ok(Self(E::G1Affine::deserialize_with_mode(
            reader, mode, validate,
        )?))
    }
}

#[cfg(test)]
mod tests {
    use super::*;
    use crate::SRS;
    use ark_bls12_381::{Bls12_381, Fr};
    use ark_ec::Group;
    use ark_poly::polynomial::univariate::DensePolynomial;
    use ark_poly::DenseUVPolynomial;

    #[test]
    fn test_commit_zero_polynomial() {
        // Zero polynomial should commit to identity (zero element)
        let g1 = <Bls12_381 as Pairing>::G1::generator();
        let g2 = <Bls12_381 as Pairing>::G2::generator();

        let srs: SRS<Bls12_381> = SRS {
            powers_of_g: vec![g1.into_affine(); 10],
            powers_of_h: vec![g2.into_affine(), (g2 * Fr::from(2u64)).into_affine()],
            max_degree: 9,
        };

        // Zero polynomial
        let poly = DensePolynomial::from_coefficients_vec(vec![Fr::from(0u32)]);

        let commitment = Commitment::<Bls12_381>::commit(&poly, &srs);

        // Should be identity (zero) element in G1
        assert!(commitment.0.is_zero());
    }

    #[test]
    fn test_commit_constant_polynomial() {
        // P(x) = 5, commitment should be 5·G
        let g1 = <Bls12_381 as Pairing>::G1::generator();
        let g2 = <Bls12_381 as Pairing>::G2::generator();

        let srs: SRS<Bls12_381> = SRS {
            powers_of_g: vec![g1.into_affine(); 10],
            powers_of_h: vec![g2.into_affine(), (g2 * Fr::from(2u64)).into_affine()],
            max_degree: 9,
        };

        // Constant polynomial P(x) = 5
        let poly = DensePolynomial::from_coefficients_vec(vec![Fr::from(5u32)]);

        let commitment = Commitment::<Bls12_381>::commit(&poly, &srs);

        // Commitment should be 5·G
        let expected = (g1 * Fr::from(5u32)).into_affine();
        assert_eq!(commitment.0, expected);
    }

    #[test]
    fn test_commit_degree_validation() {
        // Test degree validation
        let g1 = <Bls12_381 as Pairing>::G1::generator();
        let g2 = <Bls12_381 as Pairing>::G2::generator();

        let srs: SRS<Bls12_381> = SRS {
            powers_of_g: vec![g1.into_affine(); 5], // max degree 4
            powers_of_h: vec![g2.into_affine(), (g2 * Fr::from(2u64)).into_affine()],
            max_degree: 4,
        };

        // Degree 4 polynomial (should work)
        let poly = DensePolynomial::from_coefficients_vec(vec![Fr::from(1u32); 5]);
        let _comm = Commitment::<Bls12_381>::commit(&poly, &srs);

        // Degree 5 polynomial (should panic)
        let poly = DensePolynomial::from_coefficients_vec(vec![Fr::from(1u32); 6]);
        let result = std::panic::catch_unwind(|| {
            Commitment::<Bls12_381>::commit(&poly, &srs);
        });

        assert!(result.is_err(), "Should panic for degree > max_degree");
    }
}
