SRS (Structured Reference String) generation and loading

This module handles the trusted setup for the KZG10 polynomial commitment scheme.

§CRITICAL SECURITY WARNING

§The Toxic Waste Problem

The SRS generation requires a secret value s (called “toxic waste”) that:

  1. MUST be kept secret - If s is known, an attacker can:

    • Forge arbitrary polynomial commitments
    • Create fake proofs that verify correctly
    • Break the entire security of the KZG10 scheme

  2. MUST be destroyed after use - Even if s is used in a ceremony, it must be:

    • Zeroized from memory immediately after computing powers
    • Never written to disk, logs, or crash dumps
    • Never transmitted over any network

  3. MUST be generated securely - The entropy source for s must be:

    • Cryptographically secure random number generator
    • Not predictable or reproducible by attackers
    • Sufficiently large (field element, ~254 bits for BLS12-381)

§Production Requirements

In production, you NEVER generate the SRS yourself. Instead:

  1. Participate in a Multi-Party Computation (MPC) ceremony where:

    • Multiple participants contribute randomness
    • No single participant knows the final s
    • Each participant verifies the previous contribution
    • The ceremony is publicly verifiable and auditable

  2. Use a trusted ceremony output such as:

    • Ethereum’s KZG ceremony (for Proto-Danksharding)
    • Celo’s ceremony
    • Perpetual Powers of Tau ceremony

  3. Verify the ceremony transcript before using the SRS:

    • Check all powers of s are consistent
    • Verify pairing checks: e(sⁱ·G₁, H₂) = e(G₁, sⁱ·H₂)
    • Ensure the degree is sufficient for your use case

§Mathematical Background

The SRS contains powers of the secret s multiplied by group generators:

SRS = {
    G₁, s·G₁, s²·G₁, ..., sⁿ·G₁  in G₁ (for commitments)
    H₂, s·H₂                      in G₂ (for verification)
}

where:
- G₁ is generator of G₁ group (order r curve points)
- H₂ is generator of G₂ group (order r curve points)
- s is the toxic secret (scalar field element)
- n is the maximum polynomial degree supported

§Why This Structure?

  1. Commitment: To commit to polynomial P(x) = Σ cᵢ·xⁱ:

    C = Σ cᵢ · (sⁱ·G₁) = P(s)·G₁

    This is a multi-scalar multiplication (MSM) of coefficients with SRS powers.

  2. Verification: To verify P(z) = v with proof π:

    e(π, H₂ - [z]H₂) ?= e(C - [v]G₁, H₂)

    Uses pairing with only 2 G₂ elements (H₂ and s·H₂).

  3. Efficiency: We only need n+1 G₁ elements (large) but only 2 G₂ elements (small).

§Storage Requirements

For a degree n SRS on BLS12-381:

  • G₁ elements: 48 bytes each → (n+1) × 48 bytes
  • G₂ elements: 96 bytes each → 2 × 96 bytes

Examples:

  • Degree 2¹⁶ (65536): ~3.1 MB
  • Degree 2²⁰ (1M): ~50 MB
  • Degree 2²⁴ (16M): ~800 MB

§This Module

This module provides:

  1. setup_for_testing() - FOR TESTING ONLY

    • Generates SRS from a known secret
    • Feature-gated behind test-only feature
    • NEVER use in production
    • Used for unit tests and integration tests

  2. from_powers_of_tau() - FOR PRODUCTION

    • Loads SRS from a powers of tau ceremony
    • Validates all powers are consistent
    • Performs pairing checks to verify correctness
    • Safe for production use

  3. Serialization methods - Save/load SRS to disk

    • save_to_file() - Write SRS in binary format
    • load_from_file() - Read and validate SRS from file

§Example: Testing Setup (INSECURE!)

use ark_bls12_381::{Bls12_381, Fr};
use kzg10::SRS;

// WARNING: This is for TESTING ONLY!
// NEVER use known secrets in production!
#[cfg(feature = "test-only")]
{
    let secret = Fr::from(42u64);  // INSECURE! predictable
    let max_degree = 1000;

    let srs = SRS::<Bls12_381>::setup_for_testing(secret, max_degree)
        .expect("Setup failed");
}

§Example: Production Setup (SECURE)

ⓘ
use ark_bls12_381::Bls12_381;
use kzg10::SRS;

// Load from trusted ceremony output
let srs = SRS::<Bls12_381>::load_from_file("ceremony_output.srs")
    .expect("Failed to load SRS");

// Verify degree is sufficient
srs.validate_degree(1000)
    .expect("SRS degree too small");

§Security Best Practices

  1. Always use from_powers_of_tau() in production
  2. Never call setup_for_testing() in production code
  3. Enable the test-only feature only for tests
  4. Verify ceremony transcripts before using SRS
  5. Keep SRS files read-only and integrity-checked
  6. Use secure file storage with access controls
