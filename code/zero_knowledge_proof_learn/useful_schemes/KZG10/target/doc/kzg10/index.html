§KZG10 Polynomial Commitment Scheme

A production-ready implementation of the Kate-Zaverucha-Goldberg (KZG10) polynomial commitment scheme, as described in the paper: “Polynomial Commitments” by Kate, Zaverucha, and Goldberg (2010).

§Mathematical Background

The KZG10 scheme allows a prover to commit to a polynomial f(x) of degree d, and later prove evaluations of f at arbitrary points. The commitment is a single group element, and proofs are also single group elements, making the scheme extremely efficient.

§Key Properties

  • Commitment Size: O(1) - constant size regardless of polynomial degree
  • Proof Size: O(1) - constant size evaluation proofs
  • Verifier Time: O(1) - constant time verification
  • BATCH VERIFICATION: Supports efficient batch proof verification
  • Trusted Setup: Requires a one-time trusted setup ceremony

§Cryptographic Assumptions

The security of KZG10 relies on the bounded strong Diffie-Hellman assumption in groups with a pairing-friendly elliptic curve. We use BLS12-381, which provides approximately 128 bits of security.

§Protocol Overview

  1. Setup (Trusted): Generate structured reference string (SRS)

    • Powers of secret s: [1, s, s², …, sⁿ] multiplied by generator G
    • Toxic waste s must be destroyed after setup

  2. Commit: Compute commitment to polynomial f

    • Commitment = Σᵢ fᵢ · G·sⁱ where f = Σᵢ fᵢ·xⁱ
    • This is homomorphic: com(f+g) = com(f) + com(g)

  3. Open: Generate proof that f(z) = y

    • Quotient polynomial: q(x) = (f(x) - y) / (x - z)
    • Proof = commitment to q(x)

  4. Verify: Check proof using pairing product

    • e(proof, [z] - [1]₁) ?= e(commitment - [y]₂, [1]₂)
    • Uses bilinear pairing for verification

§Performance Considerations

  • Pairing Operations: Most expensive operation (~1ms on modern hardware)
  • Multi-Scalar Multiplication (MSM): Commitment uses optimized MSM (O(n log n))
  • FFT: Used for polynomial operations (O(n log n))
  • Parallelization: Feature flag parallel enables multi-threaded operations

§Architecture

This library is organized into several modules:

  • srs: Structured Reference String (trusted setup)
  • commit: Polynomial commitment generation
  • proof: Evaluation proof creation
  • verify: Proof verification (single and batch)
  • pcs: Polynomial Commitment Scheme trait and implementation
  • error: Comprehensive error types
  • fft: Optimized FFT operations

§Example Usage

ⓘ
use ark_bls12_381::Bls12_381;
use ark_poly::univariate::DensePolynomial;
use ark_poly::UvPolynomial;
use kzg10::{PCS, SRS};

// Setup: Generate SRS with toxic secret
let degree = 10;
let srs = SRS::<Bls12_381>::setup(degree, &mut rng);

// Commit: Commit to polynomial f(x) = x² + 2x + 1
let coeffs = vec![1, 2, 1]; // 1 + 2x + x²
let poly = DensePolynomial::from_coefficients_vec(coeffs);
let commitment = srs.commit(&poly)?;

// Open: Generate proof that f(5) = 36
let point = 5;
let value = poly.evaluate(&point);
let proof = srs.open(&poly, point)?;

// Verify: Check proof using only commitment
let verified = srs.verify(&commitment, point, value, &proof)?;
assert!(verified);

§Security Considerations

§Trusted Setup Requirements

The SRS generation requires a trusted setup ceremony where:

  1. Participants contribute randomness to create the secret s
  2. The toxic waste s is destroyed after setup
  3. No single participant knows s (multi-party computation)
  4. Powers of s are computed correctly: s⁰, s¹, s², …, sⁿ

If the setup is compromised (s is known), an attacker can forge proofs.

§Zeroization

The secure feature enables automatic zeroization of sensitive data:

  • Secrets are zeroized from memory when dropped
  • Prevents memory leakage attacks
  • Adds small performance overhead

Enable with: cargo build --features secure

§Constant-Time Operations

Critical operations use constant-time implementations to prevent timing attacks:

  • Field arithmetic in ark-ff
  • Pairing operations in ark-ec
  • Scalar multiplications

§Feature Flags

  • parallel: Enable multi-threaded operations using rayon

    • Recommended for high-degree polynomials (> 2¹⁶)
    • Improves commitment and proof generation speed

  • serde: Enable serialization/deserialization of commitments and proofs

    • Useful for storing proofs persistently
    • Required for network transmission

  • secure: Enable zeroization of sensitive data

    • Recommended for production use
    • Prevents memory leaks of secrets

  • docs: Enable all features for documentation generation
