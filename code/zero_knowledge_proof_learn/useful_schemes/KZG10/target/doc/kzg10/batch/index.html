Batch operations for efficient proof verification

This module implements batch verification that allows verifying multiple evaluation proofs simultaneously using random linear combination, reducing amortized cost by approximately n× for large batches.

§Mathematical Background: Random Linear Combination

Instead of verifying n proofs individually (requiring 2n pairings), we combine them using random coefficients and verify with only 2 pairings total.

§The Batch Verification Algorithm

Given n proofs (Cᵢ, zᵢ, vᵢ, πᵢ) for i = 1..n:

§Step 1: Generate Random Coefficients

Generate unpredictable random scalars r₁, r₂, …, rₙ from the field:

rᵢ ←$ {0, 1, ..., p-1} for i = 1..n

§Step 2: Compute Linear Combinations

Combine all commitments, proofs, and values using the random coefficients:

C_batch =  Σᵢ rᵢ · Cᵢ         (combined commitment)
π_batch =  Σᵢ rᵢ · πᵢ         (combined proof)
v_batch =  Σᵢ rᵢ · vᵢ         (combined value)

Note: All proofs must use the same evaluation point z.

§Step 3: Verify Single Pairing Equation

Check the aggregated pairing equation:

e(π_batch, (s - z)·H) ?= e(C_batch - v_batch·G, H)

§Why This Works: Linearity of Pairings

The pairing operation e(·,·) is bilinear, meaning:

e(a·P, b·Q) = e(P, Q)^(a·b)
e(P₁ + P₂, Q) = e(P₁, Q) · e(P₂, Q)
e(P, Q₁ + Q₂) = e(P, Q₁) · e(P, Q₂)

§Proof of Correctness

If each individual proof is valid (Pᵢ(z) = vᵢ), then:

Individual verification equations:
e(πᵢ, (s - z)·H) = e(Cᵢ - vᵢ·G, H)  for all i = 1..n

Multiply each equation by rᵢ (raise to power rᵢ in GT):

e(πᵢ, (s - z)·H)^(rᵢ) = e(Cᵢ - vᵢ·G, H)^(rᵢ)

By bilinearity, this is equivalent to:

e(rᵢ·πᵢ, (s - z)·H) = e(rᵢ·(Cᵢ - vᵢ·G), H)
e(rᵢ·πᵢ, (s - z)·H) = e(rᵢ·Cᵢ - rᵢ·vᵢ·G, H)

Multiply all equations together:

Πᵢ e(rᵢ·πᵢ, (s - z)·H) = Πᵢ e(rᵢ·Cᵢ - rᵢ·vᵢ·G, H)

By bilinearity of pairings:

e(Σᵢ rᵢ·πᵢ, (s - z)·H) = e(Σᵢ rᵢ·Cᵢ - (Σᵢ rᵢ·vᵢ)·G, H)

Which is exactly our batch verification equation:

e(π_batch, (s - z)·H) = e(C_batch - v_batch·G, H)

Therefore, if all individual proofs are valid, the batch equation is valid.

§Soundness: Why Cheating Fails

The critical question: Can a malicious prover create fake proofs that pass batch verification but fail individual verification?

Answer: No, under the q-SBDH assumption, with overwhelming probability.

§The Fiat-Shamir Heuristic

The random coefficients r₁, …, rₙ act like challenges in a Σ-protocol. If a prover could create invalid proofs that pass batch verification, they would need to “guess” the random coefficients in advance.

Since the coefficients are:

  • Generated from a cryptographically secure random source
  • Unpredictable before verification time
  • Different for each batch verification

The probability of guessing correct coefficients is 1/p ≈ 2^(-254) for BLS12-381, which is cryptographically negligible.

§Formal Security Argument

Suppose a prover creates n invalid proofs. For the batch to verify, they need:

Πᵢ e(πᵢ, (s - z)·H)^(rᵢ) = Πᵢ e(Cᵢ - vᵢ·G, H)^(rᵢ)

This is a polynomial equation in the random variables r₁, …, rₙ:

Σᵢ rᵢ · (individual_verification_failure) = 0

For this to hold for random rᵢ, all coefficients must be zero, meaning all individual proofs must be valid. This happens with probability:

Pr[batch_accept | ∃ invalid proof] ≤ 1/p

§Performance Analysis

§Computational Cost

Individual Verification (n proofs):

Cost_individual = n × (2 pairings + 1 scalar mul + 2 subtractions)
                ≈ n × 2 pairings
                ≈ n × 0.6ms (for BLS12-381)

Batch Verification (n proofs):

Cost_batch = n × (2 scalar muls) + 1 × (2 pairings) + O(n) additions
           ≈ n × 0.01ms + 0.6ms
           ≈ 0.6ms for large n

Speedup:

Speedup(n) = Cost_individual / Cost_batch
           ≈ (n × 0.6ms) / 0.6ms
           ≈ n

§When to Use Batch Verification

Use batch verification when:

  • Verifying 3+ proofs (speedup > 2x)
  • Proofs use the same evaluation point z
  • Network latency makes individual verification slow
  • You need to verify many blocks/transactions simultaneously

Use individual verification when:

  • Verifying 1-2 proofs (overhead not worth it)
  • Proofs use different evaluation points
  • You need to know which specific proof is invalid

§Security Requirements

§CRITICAL: Randomness Quality

The security of batch verification completely depends on the quality of the random coefficients r₁, …, rₙ.

§Requirements

  1. Cryptographically Secure: Use OsRng or thread_rng() from rand
  2. Unpredictable: Must not be guessable by attacker
  3. Unique: Different for each batch verification
  4. Uniform: Each rᵢ uniformly random from [0, p-1]

§What Goes Wrong If Randomness Is Bad?

Predictable randomness (e.g., rᵢ = i):

  • Attacker can craft invalid proofs that pass batch verification
  • Breaks soundness guarantee
  • Complete security failure

Insufficient entropy (e.g., only 8 bits of randomness):

  • Attacker can brute force all 2⁸ = 256 possibilities
  • 1/256 chance of finding a working combination
  • Unacceptable security level

Reused randomness (same rᵢ across verifications):

  • Attacker can learn information about proofs
  • Potential for adaptive attacks
  • Weakens zero-knowledge property

§Our Implementation

We use ark_std::rand::rngs::OsRng which:

  • Draws from operating system’s CSPRNG
  • Provides cryptographic-quality randomness
  • Is unpredictable and has sufficient entropy
  • Is the standard choice for production Rust code

§Limitations

§Same Evaluation Point

Current limitation: All proofs in a batch must use the same evaluation point z. This is because we compute (s - z)·H once and reuse it.

Workaround: For different evaluation points, group proofs by zᵢ and verify each group separately. The speedup still applies to each group.

Future extension: Can generalize to different zᵢ using:

e(π_batch, s·H - Σᵢ rᵢ·zᵢ·H) = e(C_batch - v_batch·G, H)

This requires computing Σᵢ rᵢ·zᵢ·H instead of just z·H.

§No Error Localization

Batch verification only tells you “all valid” or “at least one invalid”. It doesn’t tell you which proof(s) are invalid.

Solution: If batch verification fails, fall back to individual verification to identify the bad proof(s).
